## Status vocabulary

- **Implemented:** the baseline has a production-path implementation and focused verification.
- **Partial:** data or scheduling exists, but some advertised behavior is not applied by the production renderer/runtime.
- **Missing:** no implementation or admitted production asset was found at the anchored baseline.
- **Unverified:** a claim exists in metadata or historical evidence, but this audit did not reproduce the complete live or visual acceptance path.

## Executive finding

The authoritative Python character is a deterministic **full-pose snapshot player with procedural face and effect overlays**, not an articulated rig. Its production package contains 89 canonical 72x96 poses, but only 39 are admitted as graph clip samples. The remaining 50 are explicitly graph-classified `diagnostic_only` at capability tier C. The graph defines 28 clips, 28 nodes, 47 directed transitions, five transition recipes, five logical channel masks, and explicit facing/action fallbacks.

The strongest implemented behaviors are ground movement, a south/front-biased authored action library, front and southeast flight, deterministic pose selection, discrete eye aim, blink, seven mouth states, ten runtime expressions, the staff-bearing action set, magic/thinking overlays, and scoreless speech/music performance. The largest gaps are a single complete machine-readable capability contract, executable graph transition policy, region-addressable layering, score-driven gaze application, a real dance mapping, independent props, and admission of the 50 feelings/social poses into the Python production graph.

The character package is the correct authoritative entry point, but it is not yet a sufficient capability manifest. Detailed truth is split across the package, graph, pose library, semantic map, runtime enums, application adapter, and evidence manifests. Consumers cannot safely infer support from the current six free-form capability strings.

### Expressions, eyes, gaze, blink, and mouth

**Expressions - implemented with limits.** The exact runtime expression set is `neutral`, `happy`, `thinking`, `surprised`, `worried`, `amused`, `confident`, `focused`, `skeptical`, and `explaining`. Each definition provides brow styles, an eye style, and a default mouth. Production tests prove that expression and blink commands change reference-pose pixels.

**Eyes/blink - implemented.** The reference renderer first detects visible cool/white eye pixels near authored eye anchors. It then applies white/blue eye cells, brow pixels, and a closed-eye overlay. Rear or occluded poses with no detected eye layout receive no face overlay. Blink timing is deterministic; the reference path treats `blink_phase >= 0.965` as closed.

**Gaze - partial.** `_reference_eye_aim` supports a discrete local x aim of -1, 0, or +1. Priority is world `target_point`, then thinking/skeptical expression, then facing. The performance scheduler resolves `gaze_target`, but `PerformanceApplication.apply` does not apply it to `WizardState.target_point` or another gaze state. Therefore score-driven gaze is scheduler data only at this baseline.

**Mouth - implemented with limits.** The exact states are `closed`, `open_small`, `open_medium`, `open_wide`, `rounded`, `smile`, and `frown`. Speech fallback cycles deterministically at 10 Hz. The media application maps `rest`, `silence`, `closed`, `open`, and `wide` into this set. This is duration/time-driven mouth activity, not phoneme, viseme, amplitude, or word alignment.

### Locomotion and flight

**Ground locomotion - implemented.** The controller supports absolute/relative movement, paths, circle and figure-eight path generation, return-to-center, directional walk helpers, keyboard/gamepad control leases, acceleration/deceleration, bounds clamping, gradual eight-direction facing, and distance-derived walk phase. Direct controls use speed 1.35 or 2.2 while running; scripted movement defaults to `MovementState.speed = 1.25`.

**Flight - implemented with authored-coverage limits.** Takeoff, hover, travel, bank, ascend/descend, and landing are stateful. Flight target altitude is clamped to 0.6-3.8 for active ascent control; initial takeoff targets 1.8. Pose selection supplies front hover/flap, southeast glide/bank, southwest bank, air reaction, air celebration, staff-forward, and landing sequences. There is no authored north/east/west full flight set.

**Dance - partial scheduler support, missing character mapping.** `dance` is a legal score track/channel and reduced motion suppresses it. No dance action, Python graph node/clip, pose mapping, or `PerformanceApplication` dance application exists. Music fallback rotates among `flourish`, `staff_spin`, `celebrate`, and `reaction`; that is a groove fallback, not a declared dance capability.

### Props, attachments, and effects

- **Staff - implemented as character art/state, partial as a prop.** All admitted action/flight art commonly includes the staff; procedural fallback has `staff_state`, `staff_hand`, and `staff_tip`. The staff is embedded in each snapshot and cannot be independently swapped, hidden, constrained, or attached through a production prop API.
- **Wings - implemented as embedded art, partial as a channel.** Graph metadata names a wings channel and flight poses show wing states, but Python cells have no wing regions. Wings cannot be independently blended or disabled without selecting another complete pose.
- **Magic effect - implemented procedural overlay.** `magic_cast` and `reaction` add rotating cyan/gold sparks. The graph owns an `effect` channel for selected clips, but no independent effect asset/attachment manifest exists.
- **Thinking effect - implemented procedural overlay.** `thinking` adds three floating colored cells.
- **Other props - missing.** No inventory, sockets, collision bounds, ownership rules, or fallback policy was found for books, lecterns, microphones, scene objects, or replaceable hand props.

## Source of truth decision

### Current state

There is **no single complete machine-readable capability manifest** at `556701a`.

- `wizard_joe_character_package.json` is the authoritative package entry point and points to the production pose library and graph.
- `reference_avatar_animation_graph_v2.json` is the most accurate detailed capability/admission source: clips, nodes, tiers, classifications, fallbacks, facings, channels, transitions, and timing.
- `reference_avatar_pose_cells.json` is authoritative for actual pose existence, dimensions, anchors, metadata, and pixels.
- `models.py`, `expressions.json`, `semantic_animation_map.json`, and `PerformanceApplication` add runtime capabilities and restrictions not represented in the package or graph.
- Rust admission/evidence files are evidence sources, not production Python authority.

### Required direction

Keep `wizard_joe_character_package.json` as the sole discovery entry point. Replace its free-form capability list with, or make it point to, a versioned structured capability manifest that is generated and validated from the production graph, pose library, runtime enums/mappings, and admission evidence. A consumer must be able to query at least:

- supported clip/action/expression/mouth/gaze/locomotion/flight/prop/effect identifiers;
- facings, mobility modes, channel ownership, and whether ownership is whole-pose or independently compositable;
- pose and clip admission status, quality tier, provenance, and evidence digest;
- implemented versus metadata-only behavior for transitions and channels;
- fallback target and reason code per unsupported request;
- renderer/schema/runtime compatibility and content hashes;
- accessibility behavior and disabled-channel support;
- asset dimensions, anchor contract, load/memory limits, and preload policy.

The manifest must be derived or cross-validated in CI. Hand-maintaining a second list of capabilities would create another divergent truth source.

## Findings and risks

1. **High - package overstates queryability.** Six strings advertise broad capabilities but do not declare exact supported clips, facings, tiers, restrictions, fallbacks, or evidence. A planner can request unsupported omnidirectional flight, region layering, or social/feeling poses.
2. **High - scheduler capability exceeds application capability.** Gaze, dance, effects, stage, and locomotion can exist as resolved score channels without a corresponding production character application. Successful scheduling is not proof of visible behavior.
3. **High - graph transition metadata is non-operative.** Transition recipes, interrupt policies, holds, markers, curves, and region masks can be mistaken for runtime behavior even though the renderer performs atomic snapshot changes.
4. **High - tier C assets are easy to mis-admit accidentally.** All 89 poses are loadable and explicit pose override can display them, while only 39 are production clip samples. Asset existence must not equal capability admission.
5. **Medium - channel masks have no Python cell regions.** Body/staff/wings/mouth/effect metadata cannot support general region composition over the 277,285 unlabelled cells.
6. **Medium - fallback is partly silent.** Pose selection catches broad failures without producing structured fallback diagnostics, making degraded character behavior difficult to detect in a live connector session.
7. **Medium - props are baked into poses.** Staff and wings are visually present but not independent runtime entities. Prop continuity and swapping cannot be guaranteed.
8. **Medium - coverage is front-biased.** South/front action, run, fall, and flight coverage is strong; other directions rely on idle/walk reduction or fallbacks.
9. **Medium - mouth motion is timing-only.** It is deterministic and serviceable but cannot claim lip synchronization or transcript/phoneme alignment.
10. **Low - dual Python/Rust inventories invite confusion.** Rust has richer hard-coded clips and evidence, but production is Python. Reports and manifests must label runtime ownership explicitly.

## Recommended changes

1. Promote the existing character package to the sole machine-readable discovery root and add a structured, versioned capability contract validated against graph, pose, and runtime sources.
2. Add an admission gate that fails when a capability references a tier C/diagnostic-only pose, a missing clip, unsupported facing, absent application mapping, or unverified evidence digest.
3. Mark graph fields as either executable or descriptive. Implement transition policies only where the snapshot renderer can honor them; retain atomic cuts where blending would create invalid anatomy.
4. Add structured fallback events for graph load failure, missing pose, unsupported action/facing/channel, accessibility suppression, and application-level mapping loss.
5. Add a character application map from resolved performance channels to real Python actions, clips, poses, gaze state, locomotion, flight, and effects. Do not admit dance until such a mapping and visible evidence exist.
6. Decide whether independent staff/wing/body layering is required. If yes, author region data and verify compositing. If no, declare those channels `whole_pose` in the manifest so planners cannot assume additive control.
7. Admit tier C social/feeling poses selectively through reviewed Python clips, not wholesale. Require continuity, contact, silhouette, face readability, and fallback tests for each promotion.
8. Preserve direct-control lease authority and the connector's observer-only audio contract in every new mapping.
9. Add load-time, memory, and frame-cost budgets to admission because the canonical JSON library is large and copied before overlays.

## Rejected alternatives

- **Treat all 89 poses as production-capable.** Rejected because the graph explicitly classifies 50 as tier C `diagnostic_only` and none are clip samples.
- **Use the Rust admission ledger as the production manifest.** Rejected because the production delivery decision is Python/ASCILINE and Rust clip/application behavior differs.
- **Use only `reference_avatar_animation_graph_v2.json` as the complete manifest.** Rejected because expressions, mouth states, semantic mappings, application limitations, hashes, runtime compatibility, and prop constraints live elsewhere.
- **Use only the package's capability string array.** Rejected because it cannot answer exact support or admission questions.
- **Blend snapshots with generic cell crossfades.** Rejected because the production renderer intentionally found that partial dissolves create false limbs and facial artifacts.
- **Call scoreless music fallback a dance system.** Rejected because it cycles existing actions and has no dance clip/action mapping.
- **Advertise scheduler output as visible character support.** Rejected because several resolved channels are never applied by `PerformanceApplication`.
- **Silently promote explicit pose-showcase success to runtime admission.** Rejected because showcase bypasses normal graph selection and quality policy.

## Verification requirements for capability admission

No capability should move to **Implemented/admitted** until all applicable checks pass:

1. **Manifest validation:** package, capability contract, graph, pose library, runtime mapping, and evidence hashes cross-validate from a clean checkout.
2. **Existence and reachability:** every advertised action/clip/pose is reachable through the production Python server and not only a test helper, Rust path, or pose showcase override.
