# Rust Chatbot Multi-Agent Workflow

Workflow: Research -> Ideation -> Plan -> Execute -> Optimize -> Review

## Roles

| Role | Agent | Accountability |
|---|---|---|
| `RUNTIME` | Epicurus (`019f5c84-aca4-7953-b8ac-23a395e7268b`) | event contracts, runtime, speech, replay, telemetry |
| `MOTION` | Singer (`019f5c84-b193-7f70-83ff-e0c7e33edd51`) | performance graph, transitions, facial and secondary motion |
| `FLOW` | Lagrange (`019f5c84-b630-7053-b560-cab55fb3ba11`) | registry, gates, CI, browser QA, release evidence |
| `INT` | coordinator | hotspot wiring, gate decisions, commits, pushes, deployment |

The same three specialists review the coordinator synthesis and later review
each other's implemented interfaces. No agent may mark its own cross-lane gate
accepted.

## Status model

```text
PLANNED -> READY -> IN_PROGRESS -> HANDOFF_READY -> IN_REVIEW -> ACCEPTED
                              |-> BLOCKED -> READY
                              |-> FAILED -> READY
ACCEPTED -> REOPENED -> READY
```

Only `ACCEPTED` earns progress. A work item becomes `READY` only when all
dependencies are accepted, its base SHA matches the coordinator integration
head, and its required locks are free.

## One-writer locks

Coordinator-only hotspots are Cargo manifests/locks, `lib.rs`, `runtime.rs`,
`state.rs`, `controller.rs`, `animation.rs`, `pose.rs`, `pose_clip.rs`,
`pose_playback.rs`, `renderer.rs`, `frame_source.rs`, `hub.rs`, `server.rs`,
served `web/*`, promoted assets, central registry/tracker, and final evidence.

Agents create disjoint modules and tests in their assigned namespaces. Their
handoff states changed files, interfaces, tests, evidence, risks, and exact
hotspot wiring requested. The coordinator reviews and applies hotspot changes
serially.

## Gate record

Each gate JSON records schema version, exact Git SHA, dirty state, Rust and OS
versions, browser version, asset/graph/policy/binary hashes, required work IDs,
commands, exit codes, durations, totals, failures, skipped tests, thresholds,
artifact hashes, reviewers, rollback point, drill result, and `PASS` or `FAIL`.

## Execution rules

1. Freeze and push P0 before production chatbot edits.
2. Freeze contracts through three-way review before hotspot integration.
3. Agents work only in their allowlists and do not revert concurrent edits.
4. Focused tests run before handoff; coordinator runs full tests after wiring.
5. A failed structural check blocks every dependent item immediately.
6. Evidence is generated by Rust. No Python acceptance dependency is allowed.
7. Heavy every-frame artifacts remain ignored or external; compact manifests
   and curated contact sheets are committed.
8. Browser acceptance uses the Rust-served endpoint and real Chromium.
9. Deployment consumes an immutable SHA and verifies endpoint build identity.
10. Tracker and registry are updated at every handoff and gate transition.

## Required command floor

```bash
cargo fmt --manifest-path rust/wizard_avatar_pose_tool/Cargo.toml --all -- --check
cargo clippy --manifest-path rust/wizard_avatar_pose_tool/Cargo.toml --all-targets --all-features -- -D warnings
cargo test --manifest-path rust/wizard_avatar_pose_tool/Cargo.toml --locked --all-targets

cargo fmt --manifest-path rust/wizard_avatar_engine/Cargo.toml --all -- --check
cargo clippy --manifest-path rust/wizard_avatar_engine/Cargo.toml --all-targets --all-features -- -D warnings
cargo test --manifest-path rust/wizard_avatar_engine/Cargo.toml --locked --all-targets

cargo run --manifest-path rust/wizard_avatar_engine/Cargo.toml --locked --release --bin wizard-avatar-pose-evidence
cargo run --manifest-path rust/wizard_avatar_engine/Cargo.toml --locked --release --bin wizard-avatar-evidence -- --check-integrity
cargo run --manifest-path rust/wizard_avatar_engine/Cargo.toml --locked --release --bin wizard-avatar-performance
cargo run --manifest-path rust/wizard_avatar_engine/Cargo.toml --locked --release --bin wizard-avatar-soak
```

New chatbot contract, graph, speech, interruption, replay, browser, and release
checks are additive. Skipped or test-only parallel implementations do not count.

Supplemental and release commands are also mandatory once their work items
exist:

```bash
node --test rust/wizard_avatar_engine/web/tests/*.test.mjs
cargo test --manifest-path rust/wizard_avatar_engine/Cargo.toml --release --test chatbot_browser -- --ignored --nocapture
cargo run --manifest-path rust/wizard_avatar_engine/Cargo.toml --locked --release --bin wizard-avatar-soak -- --profile short
cargo run --manifest-path rust/wizard_avatar_engine/Cargo.toml --locked --release --bin wizard-avatar-soak -- --profile release
cargo build --manifest-path rust/wizard_avatar_engine/Cargo.toml --locked --release --bin wizard-avatar-server
```

Required CI check names are `rchat-contracts`, `rchat-rust`, `rchat-motion`,
`rchat-evidence`, `rchat-browser`, `rchat-soak`, and `rchat-release`. Every-frame
PNG/RGB output, browser traces/video, and full soak logs are external artifacts
retained 14 days for pull requests, 90 days for release candidates, and 180
days for final releases. Gate manifests, hashes, curated contact sheets, and
deployment receipts are committed permanently.

## Optimization and review

After functional execution, the three agents independently review:

- RUNTIME reviews motion contracts for deterministic ownership and ingress.
- MOTION reviews runtime output for visual intent, continuity, and recovery.
- FLOW reruns registry, scope, evidence, browser, release, and rollback checks.

The coordinator resolves findings, reruns affected gates, and requests a final
three-way scorecard. No open priority-1 or priority-2 finding may ship.
